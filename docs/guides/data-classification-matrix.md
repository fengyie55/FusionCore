# FusionData 数据分类矩阵

## 1. 目的

本文用于定义 `FusionCore` 当前应面对的数据类型、对象边界、`data/` 目录语义与术语表。

本文是 `FusionData` 的 P1 文档，不包含任何实现要求。

## 2. 数据分类矩阵

| 分类 | 它是什么 | 真相拥有者 | 是否适合长期持久化 | 更偏文件还是结构化存储 | 是否适合查询/统计/追溯 |
| --- | --- | --- | --- | --- | --- |
| 配置类数据 | profile、运行根、模块选项等配置内容 | FusionConfig | 是 | 文件优先 | 一般不作为统计主源 |
| 日志类数据 | 运行日志、操作日志、故障日志 | FusionLog | 视策略而定 | 文件优先，后续可补结构化 | 适合追溯，但当前不由 FusionData 实现 |
| 业务持久化数据 | 业务边界需要保留的记录化数据 | Domain / Scheduler / FA | 是 | 结构化优先 | 是 |
| 历史记录数据 | 状态变化、阶段流转、历史轨迹 | Domain / Scheduler / FA 派生 | 是 | 结构化优先 | 是 |
| 查询投影数据 | 面向查询/展示的派生视图数据 | Scheduler / FA / App 派生 | 是 | 结构化优先 | 是 |
| 文件工件类数据 | 配方工件、附件、导出包、报告等 | 对应业务边界 | 视业务而定 | 文件优先 | 一般不是主查询源 |
| 临时运行数据 | 当前运行周期中生成的临时数据 | 当前运行时 | 否 | 不进入长期存储 | 不适合作为长期追溯主源 |
| 导出/备份类数据 | 导出快照、备份包、离线归档 | 导出/备份流程 | 是 | 文件优先 | 适合归档，不一定适合在线查询 |

## 3. 对象边界定义

### 3.1 Persistence Record

`Persistence Record` 表示后续用于长期保存的结构化记录。

它不是 `Domain Entity` 的直接替代品，只是某类业务真相的持久化表达。

### 3.2 Query Projection

`Query Projection` 表示为查询、列表、追溯或展示准备的派生数据。

它来自写入真相的派生或汇总，不是写入真相本身。

### 3.3 File Artifact

`File Artifact` 表示以文件形式存在的工件。

它强调完整交付、人工可读、复制归档或格式保真，不强调结构化检索。

### 3.4 Snapshot Record

`Snapshot Record` 表示某个时点的静态保存结果。

它可以来源于真相、投影或运行态汇总，但不等于持续可变的真相对象。

### 3.5 History Record

`History Record` 表示用于记录“曾经发生过什么”的结构化历史项。

它更偏时间序列、阶段演化和对象轨迹，而不是当前状态本身。

### 3.6 Export Artifact

`Export Artifact` 表示用于导出、交换、归档或交付的结果工件。

它不一定适合作为系统的在线查询主源。

## 4. 存储边界与查询边界

### 4.1 存储边界解决什么问题

存储边界只解决以下问题：

- 哪类数据需要被长期保存
- 保存结果应偏向文件还是结构化记录
- 持久化对象应采用什么边界语义

### 4.2 查询边界解决什么问题

查询边界只解决以下问题：

- 哪类已保存或已投影数据需要被读取
- 查询应面向记录还是投影
- 查询结果是否适合列表、追溯或统计

### 4.3 当前阶段为什么只定义不实现

原因如下：

- 当前治理阶段 `FusionData` 仅为 `P1`
- 平台其他底座模块仍在持续收敛
- 过早选型数据库、文件格式或查询框架，会反向绑死后续边界
- 当前更需要清晰的职责说明，而不是提前搭建基础设施

### 4.4 当前阶段为什么不引入 Repository / ORM / Query Framework

原因如下：

- 这些都属于实现机制，而不是 P1 语义边界
- 当前并不知道最终哪些数据真的需要结构化持久化
- 当前也不应把 `FusionData` 扩成“大而全数据平台”

## 5. runtime root 下 data/ 目录语义

### 5.1 data/ 存放什么

`data/` 用于承接运行时需要保留的业务数据、历史记录、查询投影、文件工件和导出类结果。

### 5.2 data/ 不应存放什么

`data/` 不应存放：

- 配置文件
- 日志文件
- 仅在当前进程存在的临时运行数据
- 构建输出或发布脚本内容

### 5.3 与其他目录的区别

- `config/`：配置输入与模板
- `logs/`：日志输出
- `runtime/`：进程私有或运行期状态文件
- `temp/`：可丢弃的临时数据
- `backups/`：备份结果
- `data/`：业务相关的长期数据、投影、工件与导出结果

### 5.4 data/ 下的逻辑子目录建议

以下目录只是逻辑语义，不绑定唯一物理路径：

- `records/`：长期保存的结构化记录
- `projections/`：面向查询的投影结果
- `exports/`：导出结果与交换结果
- `artifacts/`：文件工件
- `snapshots/`：快照结果
- `cache/`：如后续需要，也只能表示“可重建、非真相”的缓存语义

### 5.5 物理路径要求

上述目录是逻辑分类，不得被硬编码为唯一磁盘路径。

默认逻辑运行根可以映射到 `R:\`，但不能把 `R:\` 写死为唯一前提。

## 6. 术语表

### Persistence

`Persistence` 指数据被长期保存并可在后续运行中再次读取的能力语义。

### Projection

`Projection` 指从真相或记录派生出的只读查询表达。

### Snapshot

`Snapshot` 指某个时点的静态保存结果，不等于持续可变的真相对象。

### Artifact

`Artifact` 指以文件或交付物形式存在的结果对象，强调交付、保留和可复制。

### History Record

`History Record` 指用于追踪对象或流程历史变化的记录项。

### Query Model

`Query Model` 指面向读取、列表、展示或统计的只读模型。

### Data Store

`Data Store` 指承载已保存数据的边界概念，不预设实现技术。

### File Artifact

`File Artifact` 指以文件为主表达形式的工件对象。

### Structured Store

`Structured Store` 指支持字段化记录、索引、查询或关联能力的数据存放边界概念。

## 7. 当前阶段不做什么

当前阶段明确不做：

- 数据库实现
- 文件系统实现
- 查询服务实现
- 导出系统实现
- 迁移、缓存、同步、归档平台
- 任何代码骨架扩写

## 8. 进入 P2 时最可能的方向

进入 `P2` 后，最可能的推进方向是：

- 先把“记录 / 投影 / 工件 / 快照 / 历史”的边界固化成极薄契约
- 再区分文件型边界和结构化边界
- 最后才选最小实现路径

当前阶段不应跳过边界定义，直接进入技术实现。
