# FusionStudio 信息架构

## 1. 设计目标

`FusionStudio` 的信息架构目标是：在不引入业务控制逻辑的前提下，为设备厂家和调试工程师提供一个更像 PLC 工程软件或设备工程配置软件的工作台。

它不是运行 HMI，也不要求遵循 E95。它优先考虑：

- 快速了解整机结构
- 快速定位模块
- 快速进入模块级配置、报警、互锁、IO 和调试入口
- 快速查看运行与日志摘要

## 2. 首页原则

打开后首先看到的是设备总览，而不是普通菜单页。

首页至少应展示：

- 设备实例基本信息
- Profile 与 Runtime Root 摘要
- 模块总览
- 模块状态
- 工程入口摘要
- 工程树中一级工具节点摘要

## 3. 最小布局分区

建议采用四区布局：

- 顶部：设备与工程总览区
- 左侧：工具域导航区
- 中央：主工作区
- 右侧：工程树与上下文摘要区
- 底部：状态与调试摘要区

## 4. 最小导航结构

一级导航按工具域组织：

- 设备总览
- 工程配置
- 报警配置
- 互锁管理
- 模块工作台
- IO 监控
- 运行诊断
- 详细日志
- 工程控制台
- 调试助手

## 5. 工程树原则

`FusionStudio` 不应只停留在平面页面列表，而应显式表达一棵工程树：

- 设备根节点
- 模块节点
- 模块工具节点

当前阶段的最小层级为：

- 设备
  - 模块
    - 参数
    - IO
    - 报警
    - 互锁
    - 状态
    - 调试

## 6. 模块工作台原则

工作区应支持“整机级”和“模块级”两种视角：

- 选中设备总览：显示整机摘要、模块树和入口概览
- 选中模块工作台相关入口：显示模块级工具占位

模块级页面后续可自然扩展：

- 参数
- IO
- 报警
- 互锁
- 状态
- 调试

建议在模型层显式表达：

- 设备总览模型
- 模块节点模型
- 工具节点模型
- 工程树模型

## 7. 与接线边界的关系

`FusionStudio` 页面数据应来自：

- `FusionApp` 的 bootstrap 结果
- `FusionKernel` 的宿主与模块摘要
- `FusionConfig` 的工程配置摘要
- `FusionLog` 的日志摘要

当前阶段不允许页面直接持有后台内部真相对象。

## 8. 当前阶段不做什么

当前阶段不做：

- 真实配置编辑
- 真实日志浏览系统
- 真实工程控制执行
- 复杂调试工具链
- 数据库存储与历史查询

## 9. 后续进入下一阶段的方向

建议后续优先推进：

1. 从 `FusionApp` 接入默认 bootstrap
2. 从 `FusionKernel` 接入宿主与模块摘要
3. 从 `FusionConfig` 接入工程配置摘要
4. 从 `FusionLog` 接入日志入口摘要
5. 保持“只读为主、控制后置”的演进方式

## 10. 当前阶段新增的工具页上下文

本阶段开始，工程工具页不再只拿模块列表占位，而是统一消费：

- 模块上下文
- 工具页上下文

这意味着：

- 报警配置、互锁管理、IO 监控、工程控制台页面会显示统一的上下文头
- 页面能够明确显示“当前设备 / 当前模块 / 当前工具域”
- 但页面仍然是只读骨架，不承担真实编辑和真实控制执行
