# FusionData 存储约定

## 1. 目的

本文用于说明 `FusionData` 在 P1 阶段对“文件型存储”和“结构化持久化”的分工原则。

本文只定义方向，不定义实现。

## 2. 判断规则

判断某类数据更适合文件还是结构化存储时，优先看以下因素：

### 2.1 更偏向文件型存储的场景

- 需要人工查看、复制、归档或交付
- 更像工件而不是记录
- 以完整对象为单位保存，不强调跨记录关联查询
- 后续更常见的是下载、导出、备份，而不是统计分析
- 保持原始内容或格式比结构化检索更重要

### 2.2 更偏向结构化持久化的场景

- 需要按字段检索、关联、过滤、排序或统计
- 需要形成历史记录、状态演变记录或投影查询源
- 数据量会持续增长，且查询维度较明确
- 需要支持按时间、对象 Id、状态等条件追溯
- 更适合作为查询主源，而不是人工阅读工件

### 2.3 不应直接进入长期持久化的场景

- 只在当前进程或当前运行周期中短暂存在
- 可由其他真相重新计算或重新生成
- 只用于运行时协调、临时缓存、临时导入/导出中转
- 生命周期与进程或会话强绑定

## 3. 文件 vs 结构化存储对照表

| 数据类型 | 真相拥有者 | 长期持久化 | 文件优先 | 结构化优先 | 查询/统计/追溯 |
| --- | --- | --- | --- | --- | --- |
| 配置类数据 | FusionConfig | 是 | 是 | 否 | 一般不作为统计主源 |
| 日志类数据 | FusionLog | 视策略而定 | 是 | 可选 | 可追溯，但不由 FusionData 当前阶段实现 |
| 业务持久化数据 | Domain / Scheduler / FA | 是 | 否 | 是 | 是 |
| 历史记录数据 | Domain / Scheduler / FA 派生 | 是 | 可导出 | 是 | 是 |
| 查询投影数据 | Scheduler / FA / App 派生 | 是 | 否 | 是 | 是 |
| 文件工件类数据 | 对应业务边界 | 视业务而定 | 是 | 否 | 通常不是主查询源 |
| 临时运行数据 | 当前运行时 | 否 | 否 | 否 | 不应作为长期数据源 |
| 导出/备份类数据 | 导出/备份流程 | 是 | 是 | 否 | 更偏人工恢复与归档 |

## 4. 典型分类建议

### 4.1 优先采用文件型存储的数据

- 可人工查看的工艺文件、配方工件、报告、导出包
- 快照式备份工件
- 需要保留原始格式的附件或外部文件

### 4.2 优先采用结构化持久化的数据

- 设备运行历史记录
- 批次、物料、载具、作业等业务记录
- 调度过程产生的可追溯状态记录
- 自动化投影形成的查询数据

### 4.3 只应保留在 runtime / temp 语义的数据

- 临时计算结果
- 临时中转文件
- 一次性导入上下文
- 可重建的缓存样式数据

### 4.4 适合导出/备份但不一定适合作为查询主存储的数据

- 审计导出包
- 快照归档文件
- 对外交换的工艺或报表工件

## 5. 关于“人工可读工件”和“结构化查询源”

### 5.1 人工可读工件

人工可读工件强调：

- 可直接打开查看
- 可复制、归档、传递
- 更关注完整性和可读性

它们通常更适合文件型存储。

### 5.2 结构化查询源

结构化查询源强调：

- 可按字段检索
- 可按对象关系聚合
- 可支撑列表、统计、追溯

它们通常更适合结构化持久化。

## 6. 当前阶段不做什么

当前阶段不做以下内容：

- 不选型数据库
- 不选型文件格式
- 不实现文件写入
- 不实现结构化存储
- 不实现查询引擎
- 不实现 Repository / ORM / Migration

## 7. P2 进入方向

`FusionData` 进入 P2 时，应优先按本指南的判断规则，把未来实现拆成两条边界：

- 文件工件边界
- 结构化记录 / 查询边界

而不是先选具体技术再反推边界。
