# FusionData 边界说明

## 1. 模块定位

`FusionData` 是 `FusionCore` 的数据管理与存储边界模块。

当前治理阶段为 `P1`，本阶段只定义以下内容：

- 数据分类语义
- 持久化与查询边界
- 文件型存储与结构化持久化的职责划分
- 数据对象边界与术语
- 与其他模块的关系

当前阶段明确不做：

- 数据库实现
- 文件读写实现
- ORM / Repository / Unit of Work
- 缓存、迁移、同步、归档、查询平台
- 任何真实落库、落盘、导出、导入机制

## 2. FusionData 解决什么问题

`FusionData` 解决的是“哪些数据需要被保存、查询、投影、导出，以及这些能力应由什么边界承接”的问题。

它负责定义：

- 数据是否需要长期保存
- 数据更适合文件型存储还是结构化存储
- 数据对象应以记录、投影、快照还是工件的方式表达
- 读取与查询应暴露什么边界，而不是直接把业务真相暴露给外部模块

## 3. FusionData 不解决什么问题

`FusionData` 不解决以下问题：

- 配置定义、配置加载、Profile 规则
- 日志记录、日志分类、日志输出
- 领域真相、领域规则、领域事件
- 调度算法、恢复算法、自动化协议逻辑
- UI 页面查询交互与表现层逻辑
- 宿主运行时、进程管理、IPC、消息总线

## 4. 与其他模块的边界

### 4.1 与 FusionConfig 的边界

- `FusionConfig` 负责配置语义、配置节边界、运行根与 profile。
- `FusionData` 不拥有配置真相，也不实现配置读写。
- 配置类数据只在 `FusionData` 文档中作为“数据分类样本”出现，用于说明其不归 `FusionData` 实现。

### 4.2 与 FusionLog 的边界

- `FusionLog` 负责日志语义、日志条目、日志上下文和最小写入边界。
- `FusionData` 不负责日志采集、日志持久化实现、日志查询平台。
- 日志类数据只在 `FusionData` 中作为边界说明，不代表 `FusionData` 接管日志系统。

### 4.3 与 FusionDomain 的边界

- `FusionDomain` 拥有领域真相。
- `FusionData` 不拥有领域实体、领域值对象、领域事件的真相。
- 后续若存在持久化记录，它们也只是对领域真相的存储表达，不是领域实体的替代品。

### 4.4 与 FusionScheduler 的边界

- `FusionScheduler` 拥有调度协作语义、上下文刷新语义、评估与计划骨架语义。
- `FusionData` 未来只为调度相关持久化记录、历史记录、查询投影提供边界。
- `FusionData` 不参与调度决策和执行。

### 4.5 与 FusionFA 的边界

- `FusionFA` 拥有自动化投影、外部接口映射和工厂自动化语义。
- `FusionData` 未来只为 FA 的历史记录、接口投影快照或可查询数据边界提供支撑。
- `FusionData` 不实现 SECS/GEM、HSMS 或对外交互逻辑。

### 4.6 与 FusionUI / FusionApp 的边界

- `FusionUI` 是表现层，`FusionApp` 是应用装配层。
- `FusionData` 不负责 UI 查询交互、不负责应用级业务编排。
- `FusionUI / FusionApp` 后续只能消费 `FusionData` 暴露的只读投影或查询边界，而不是直接以 `FusionData` 作为业务真相来源。

## 5. 数据拥有权原则

FusionCore 中的数据必须遵循“真相拥有者唯一”的原则：

- `FusionConfig` 拥有配置真相
- `FusionLog` 拥有日志语义与日志记录真相
- `FusionDomain` / `FusionScheduler` / `FusionFA` 拥有业务真相
- `FusionData` 不拥有业务真相，只定义存储与读取边界

因此：

- 持久化记录不是业务真相
- 查询投影不是业务真相
- 文件工件不是业务真相
- 快照只是在某个时点对真相或投影的派生表达

## 6. 当前阶段的核心输出

`FusionData` 当前 P1 的交付目标只有三类：

- 文档化的模块职责与边界
- 文档化的数据分类与文件/结构化存储分工
- 文档化的对象边界、术语表和 runtime `data/` 目录语义

任何代码骨架、项目占位、测试占位都不代表 `FusionData` 已进入 P2。

## 7. 当前阶段不做什么

当前阶段明确不做：

- 数据访问接口代码
- 查询接口代码
- 文件工件存储接口代码
- 结构化存储接口代码
- 查询模型代码
- 记录模型代码
- Snapshot / Projection / History 的代码类定义

如果仓库中存在 `FusionData` 项目占位或测试占位，应视为“超前于当前治理阶段的准备动作”，后续进入 P2 时再决定是否继续沿用。

## 8. 进入 P2 时最可能的方向

当 `FusionData` 进入 P2 时，最可能的推进方向应是：

- 先固化术语与边界对应的极薄代码契约
- 再定义文件型工件与结构化记录的最小抽象
- 再建立最小查询边界
- 最后才讨论具体技术选型

在那之前，不应提前绑定数据库、ORM、Repository 或文件格式。
